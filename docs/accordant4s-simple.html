The Problem

Normal tests tangle two things

❓ What is correct?

"After a deposit of 50, the balance should be 50." This is your oracle — the source of truth.

🎲 What to try?

"Create → deposit → withdraw → deposit again." This is your sequence — which paths to exercise.

In a normal test, both are welded together

Want a different path? Rewrite the test. Want a different oracle? Rewrite every test. The result: coverage holes you can't see, and brittle suites that test implementation details rather than behavioural contracts.

Analogy: a restaurant critic who can only review one fixed meal. Separate the standard from the sampling, and you can review any meal against the same standard.

The Core Idea

Separate the oracle from the sequence

Spec[S]
the oracle ← allows → (request, response)
observed → Verdict[S]
Conformant or Deviant

Once the oracle is a value you can call — spec.allows(op, req, res, profile) — any sequence source can be checked against it:

🔍 Explore

Find reachable states from a finite input set

📋 Generate

Produce test cases with state/transition coverage

🌐 Replay

Run against a real system and judge every response

The Running Example

A bank with accounts and balances

The state is a map of account IDs to balances. Each operation has a behaviour (the oracle rule) and a mock (for offline exploration).

// The state: account id → balance (from BankState.scala) final case class BankState(accounts: Map[String, BigDecimal]) derives CanEqual // Request and response types (from InputFixtures.scala) final case class DepositRequest(accountId: String, amount: BigDecimal) final case class DepositResponse(newBalance: BigDecimal) // The deposit operation: oracle rule + mock response generator val deposit: Operation[DepositRequest, DepositResponse, BankState] = Operation( name = OperationName("Deposit"), behaviour = (req, _) => expect((_: DepositResponse) => ().validNel[SpecViolation]) .thenState((res, s) => s.copy(accounts = s.accounts.updated(req.accountId, res.newBalance)) ), mock = (req, s) => Gen.constant( DepositResponse(s.accounts.getOrElse(req.accountId, BigDecimal(0)) + req.amount) ) )

behaviour: (Req, S) ⇒ Outcome

The pure oracle rule: what response is correct, and how the state transitions. No side effects — just logic.

mock: (Req, S) ⇒ Gen[Res]

Supplies realistic responses during offline exploration. Bypassed when replaying against a real system.

What Gets Tested

spec.allows — one method, one verdict

The oracle doesn't care how the response arrived — it only asks: "given this request and this state, is this response correct?"

// The oracle method (from Spec.scala) def allows[Req, Res](op: Operation[Req, Res, S], req: Req, res: Res, profile: StateProfile[S])(using StateOps[S]): Verdict[S] = if !operations.contains(op.name) then Verdict.Deviant(NonEmptyList.one(SpecViolation.UnknownOperation(op.name))) else ProfileEval.allows(op.name, s => op.behaviour(req, s), res, profile) // A real test (from OracleCoreProperties.scala) val profile = StateProfile.one(BankState(Map("alice" -> BigDecimal(50)))) fullSpec.allows( withdraw, WithdrawRequest("alice", BigDecimal(30)), WithdrawResponse.Success(BigDecimal(20)), profile ) match case Verdict.Conformant(_) => Result.success case Verdict.Deviant(_) => Result.failure

✓ Conformant

The response passes the oracle's check. The state transitions forward. The surviving state is carried to the next step.

Example: withdraw 30 from alice (balance 50) → response says newBalance=20. Oracle says: correct. State becomes {alice → 20}.

✗ Deviant

The response fails the oracle's check. Every failed atomic check is accumulated — not just the first. Structured as NonEmptyList[SpecViolation].

Example: withdraw from unknown account → response says Success(-10). Oracle says: wrong. Deviant with accumulated violations.

How It Works

The pipeline: model → explore → generate → replay

Spec[S] + InputSet
the model + what to try
↓
GraphExplorer
BFS over mock responses
↓
StateGraph[S]
all reachable states + edges
↓
TestCaseGenerator
state/transition coverage
↓
Vector[TestCase[S]]
sequences to replay
↓
TestCaseExecutor
replay against real SUT
↓
ExecutionReport
Passed | DeviatesAt(step, violations)

Every edge in the graph is oracle-conformant by construction. The executor replays against the real system and validates every response through the same oracle.

The Bank Example

Withdraw: non-determinism as data

A withdraw might succeed (balance decreases) or return not found (account doesn't exist). The oracle doesn't know which until it sees the response — so it keeps both branches alive.

// Withdraw: OneOf(success | notFound | timeout) — from GraphFixtures.scala val withdraw: Operation[WithdrawRequest, WithdrawResponse, BankState] = Operation( withdrawName, (req, _) => expect.oneOf( NonEmptyList.of[Outcome[WithdrawResponse, BankState]]( // branch 1: success — balance decreases expect(wSuccess).thenState((res, s) => res match case WithdrawResponse.Success(b) => s.copy(accounts = s.accounts.updated(req.id, b)) case _ => s ), // branch 2: not found — state unchanged expect(wNotFound).sameState, // branch 3: timeout — state unchanged expect(wTimeout).sameState ) ), (req, s) => s.accounts.get(req.id) match case Some(bal) if bal >= req.amount => Gen.constant(WithdrawResponse.Success(bal - req.amount)) case _ => Gen.constant(WithdrawResponse.NotFound) )

What happens at runtime

Offline: the mock decides the response based on balance → BFS records the correct transition.
Replay: the SUT returns the actual response → spec.allows validates it against all branches. If the SUT says Success(-1) on an empty account, the success-check fails (negative balance), the notFound-check fails (wrong variant), and the timeout-check fails too → Deviant with three accumulated violations.

Why This Matters

The oracle doesn't just test — it judges

One spec, many uses

The same Spec[S] drives graph exploration, test generation, replay against a real system, and linearizability checking. Write the oracle once; consume it everywhere.

No drift between model and tests

The oracle is the test. There's no separate assertion file to maintain. When the contract changes, the spec changes — and every consumer automatically picks up the new rules.

Formally verified kernel

The survivor/verdict algorithm is proved correct with Stainless (9/9 verification conditions). The oracle that judges your system is itself verified — stronger than any test suite.

Contract-derived specs

Derive Operation slots directly from a Smithy IDL. A new operation breaks the spec build until the rule is written — the oracle cannot silently drift from the contract.

Separate the standard from the sampling.

The oracle knows what "correct" means. Everything else is just choosing what to try.

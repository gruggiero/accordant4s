The Problem with Conventional Tests

Tests tangle two questions

❓ What is correct behaviour?

The assertions: "after a deposit of 50, the balance should be 50." This is your oracle — the source of truth for what "right" means.

🎲 Which sequences to try?

The test cases: "create → deposit → withdraw → deposit again." This is your sequence source — which paths through the system to exercise.

The entanglement

In a normal test, both are welded together: you write one sequence with one set of assertions baked in. Want to try a different path? Rewrite the test. Want a different oracle? Rewrite every test. The result: coverage holes you can't see, and brittle suites that test implementation details rather than behavioural contracts.

Analogy: a restaurant critic who can only review one fixed meal. Separate the standard (what good food is) from the sampling (which dishes to try), and suddenly you can review any meal against the same standard.

The Core Idea

Separate the oracle from the sequence source

Once the oracle is a value you can call — spec.allows(op, req, res, profile) — any sequence source can be checked against it:

🔍 BFS Exploration

Explore reachable states from a finite input set

📋 Test Generation

State/transition coverage, random walks

🌐 Real SUT Replay

Replay against HTTP, in-memory, any system

The verdict — "deviates at step N: expected X, got Y, path P" — is actionable semantic feedback, not a generic assertion failure. Especially useful for validating AI-generated implementations.

The Model

A Spec is a typed state machine

You describe your system as operations that transform state. Each operation has a name, a behaviour (the oracle rule), and a mock (for offline exploration).

// The bank: accounts with balances final case class BankState(accounts: Map[String, BigDecimal]) val deposit: Operation[DepositRequest, DepositResponse, BankState] = Operation( name = OperationName("Deposit"), behaviour = (req, _) => expect((_: DepositResponse) => ().validNel) .thenState((res, s) => s.copy( accounts = s.accounts.updated(req.accountId, res.newBalance) )), mock = (req, s) => Gen.constant( DepositResponse(s.accounts.getOrElse(req.accountId, 0) + req.amount) ) )

behaviour: (Req, S) ⇒ Outcome

The pure oracle rule: what response is correct, and how the state transitions. No side effects.

mock: (Req, S) ⇒ Gen[Res]

Supplies realistic responses during offline BFS exploration. Bypassed when replaying against a real system.

The Core ADT

Outcome — three ways a response can be correct

enum Outcome[Res, S]: case Same(check: ResponseCheck[Res]) case Next(check: ResponseCheck[Res], transition: (Res, S) => S) case OneOf(branches: NonEmptyList[Outcome[Res, S]])

OneOf: non-determinism as data

A Withdraw might succeed (balance decreases) or not found (account doesn't exist). The oracle doesn't know which until it sees the response. So it keeps both branches alive as candidate states — a state profile.

// Withdraw: either Success OR NotFound expect.oneOf( expect(successCheck).thenState((res, s) => s.copy(...)), expect(notFoundCheck).sameState )

Same

Check passes, state unchanged. E.g. a read-only GetAccount.

Next

Check passes, state advances via a response-dependent transition.

OneOf

Non-deterministic: union of every passing branch. Models timeouts, races.

The Heart of the Library

spec.allows — one method, one verdict

def allows[Req, Res]( op: Operation[Req, Res, S], req: Req, res: Res, profile: StateProfile[S] ): Verdict[S] = if !operations.contains(op.name) then Verdict.Deviant(NonEmptyList.one(SpecViolation.UnknownOperation(op.name))) else ProfileEval.allows(op.name, s => op.behaviour(req, s), res, profile)

✓ Conformant

Returns the surviving profile — the deduplicated set of next-states from every passing branch. This threads forward to the next step.

✗ Deviant

Returns every failed atomic check, accumulated flatly — not just the first. Structured as NonEmptyList[SpecViolation].

That's it. Six lines of code. Everything else — exploration, generation, execution, linearizability — is a consumer of this one method.

Non-determinism as Data

State profiles track indefinite failures

What happens when a timeout may or may not have taken effect? The oracle doesn't know — so it keeps both candidate states alive.

timeout
may have fired → StateProfile
{ balance=100, balance=0 } → later observation
collapses to one
// A StateProfile is a non-empty, Eq-deduplicated set of candidate states opaque type StateProfile[S] = NonEmptyList[S] // The oracle evaluates ALL candidates, keeps the survivors: val survivors = candidates.flatMap(c => branches.filter(_.matches(res, c)).map(_.next(res, c)))

Why this matters

Accordant calls this "indefinite failure" — a timeout you can't immediately resolve. Instead of guessing, the oracle forks the state and carries both possibilities forward. A later observation (e.g. a successful read) collapses the profile to the surviving branch. No false alarms, no missed races.

Worked Example

The bank: withdraw with branching

val withdraw: Operation[WithdrawRequest, WithdrawResponse, BankState] = Operation( name = OperationName("Withdraw"), behaviour = (req, _) => expect.oneOf(// two possible outcomes: // branch 1: success — balance decreases expect(res => res.newBalance >= 0).thenState((res, s) => s.copy(accounts = s.accounts.updated(req.id, res.newBalance)) ), // branch 2: not found — account doesn't exist expect(_ => WithdrawResponse.NotFound).sameState ), mock = (req, s) => s.accounts.get(req.id) match case Some(bal) if bal >= req.amount => Gen.constant(Success(bal - req.amount)) case _ => Gen.constant(NotFound) )

What happens at runtime

Offline exploration: the mock decides the response based on balance → BFS records the correct transition. Real replay: the SUT returns the actual response → spec.allows validates it against both branches. If the SUT says Success(-1) on an empty account, the success-check fails (newBalance < 0) AND the notFound-check fails (wrong variant) → Deviant with two accumulated violations.

Integration: HTTP

Transport outcomes are data, never exceptions

A deployed service becomes a SystemUnderTest[IO, S] via http4s. But timeouts and connection failures don't throw — they surface as response values the oracle can reason about.

enum TransportOutcome: case Completed(status: Status, body: String) case TimedOut case ConnectionFailed(detail: String) // The mapper turns transport facts into domain responses: val sut = Http4sSut(client, binding, timeout, retries) // sut: SystemUnderTest[IO, S] — execute returns IO[call.Res]

Why this matters for the oracle

Accordant's indefinite-failure modeling requires the oracle to SEE timeouts as responses. If transport errors threw, Outcome.OneOf(timeout→Same, timeout→Next) would be untestable over HTTP. By making them data, the oracle can model "the timeout may or may not have taken effect" exactly as it models any other non-deterministic response.

Integration: Smithy IDL

The contract IS the oracle

Derive typed Operation slots directly from a smithy4s Service — so the behavioural spec and the implementation share the Smithy IDL as a single source of truth. No drift.

TestBank.smithy
the IDL → smithy4s codegen
Service[Alg] → SmithyOps.forService
EndpointSlot per op → SpecBuilder.build
Spec[S] — complete or fail
// Derive slots from the generated service, assign behaviours, build the spec val slots = SmithyOps.forService(TestBankGen) val spec = SpecBuilder[BankState](slots) .assign(withdrawName, withdrawBehaviour, withdrawMock) .assign(depositName, depositBehaviour, depositMock) .build // Right(Spec) — or Left(missing operation names)

Complete-or-fail

A new Smithy operation breaks the spec build until the rule is written. The oracle cannot silently drift from the contract — that's the whole point of derivation.

Concurrency

Linearizability: is there a sequential explanation?

Concurrent test cases run operations in parallel, then ask: does some sequential ordering explain the observed responses?

prefix
CreateSlot("9am") → parallel
BookSlot(alice) ∥ BookSlot(bob) → suffix
GetSlot("9am")

✓ Linearizable

Alice→200, Bob→409. Explained by: Alice booked first, Bob got a conflict. The checker finds the witness ordering [Alice, Bob] by brute-force search over ≤ 4! = 24 permutations.

✗ Race Detected

Alice→200, Bob→200. No sequential ordering explains both succeeding on the same slot. The checker returns None → RaceDetected with a persistable repro case.

The permutation search is a pure function — no IO, no randomness. It folds spec.allows over each candidate ordering, threading the state profile. Bounded by ParallelWidth ≤ 4 (Iron type constraint, not a runtime guard).

Beyond Testing

The kernel is formally verified

The survivor/verdict algorithm isn't just tested — it's proved correct with Stainless, a formal verifier for Scala.

🔬 The Stainless mirror

A 70-line PureScala mirror of the oracle kernel (OracleKernel.scala) runs on a dedicated Scala 3.7.2 module. Stainless proves 9/9 verification conditions — including "conformant iff some branch passed" and "the survivor set is deduplicated."

🔗 The bridge test

How do we know the verified mirror matches production? A property test (OracleModelBridgeTests) runs the real ProfileEval.allows and the Stainless OracleKernel.survivors on identical generated inputs and asserts they agree. Drift fails CI.

Why this is a big deal

Accordant (.NET) has runtime-only checking. accordant4s has a formally proved kernel — stronger than any test suite. The oracle that judges your system is itself verified. And the oracle is pure (no IO, no side effects), which is what makes it verification-eligible.

Kernel preamble

What a kernel is in this project, and why this one matters

In this codebase, a kernel is the smallest pure computation that carries the correctness burden of a feature. It is the algorithmic core that turns inputs into a decision before adapters, type wrappers, mocks, or integration details get involved. For the oracle, the kernel is the logic that answers a single question: given a response and a set of candidate states, which next states still survive?

That kernel works in three conceptual moves. First, inspect every candidate-state branch combination and keep only the branches whose checks accept the observed response. Second, compute the next state each passing branch implies. Third, deduplicate those next states and decide the verdict from emptiness: non-empty means there is still at least one consistent world, so the response is conformant; empty means every branch was ruled out, so the response is deviant.

verified/OracleKernel.scala isolates exactly that core. It is not a second product implementation; it is a deliberately tiny mirror of the shipped algorithm, reduced to a form Stainless can reason about. The production code keeps the richer domain model, while the verified mirror keeps only the proof-carrying shape.

Production kernel

Spec.allows delegates to ProfileEval.allows, which uses OutcomeEval.flatten, branch checks, transitions, and StateProfile.

Verified kernel

EvalBranch(passed, next) strips that logic down to the observable effect of each branch against a fixed response.

Why not verify production directly?

Because Cats validation, Iron, opaque types, and function-heavy state transitions are much harder for Stainless to model than a tiny PureScala mirror.

How assurance is layered now

Ring 3 checks the oracle spec, Ring 5 hardens direct public kernel behavior, and the new Ring 6 bridge cross-checks production and verified survivor logic on the same generated inputs.

Rule of thumb: the oracle kernel is the part that turns “many possible states” into “these are the survivors”. Everything else either feeds that computation or packages its result.
Saved explanation 1

What the file is trying to do

OracleKernel.scala is not the shipped oracle implementation. It is a deliberately reduced model whose only job is to make the core control-flow of the oracle simple enough for Stainless to prove. In the real code, the oracle receives a request, a response, and a non-empty profile of candidate states. It then asks: which branches of the specification accept this response, what next states do they produce, and after deduplicating those states, is anything left?

The verified file compresses all of that down to the observable essence of a branch: did it pass, and if it did, which next-state identity survives? That reduction is why the model can use EvalBranch(passed, next) over BigInt instead of the full Outcome tree, response checks, transitions, opaque state profiles, and Cats-based validation used in core.

Primary goal

Prove the oracle's conformance boundary: survivors are non-empty exactly when at least one branch passed.

Why that matters

The entire verdict split depends on this boundary. Non-empty survivors mean conformant; empty survivors mean deviant.

What is left to tests

Full survivor provenance and richer dedup semantics stay in Ring 3 property tests because those proofs are heavier and less solver-friendly.

What changed in the codebase

New tests now check both the production kernel directly and the mechanical bridge from production evaluation to the Stainless model, so the explanation is no longer just conceptual.

passingNexts: extract the next states from passing branches

This is the verified stand-in for the production pattern filter(_.matches).map(_.next). It walks the list of evaluated branches and keeps b.next only when b.passed is true. Semantically, it says that failing branches contribute nothing and each passing branch contributes one candidate survivor. The file comments note that the strongest soundness theorem, "every produced state comes from some passing branch", is intentionally not encoded here because the required quantifier instantiation makes Z3 diverge.

distinct: deduplicate while preserving first occurrence

This function keeps the head, removes later copies of it, and recurses. The decreases(xs.size) annotation is there for termination: Stainless must see a strictly shrinking measure. Conceptually, the function models the real oracle's collapse of duplicate next-states, where multiple passing branches may represent the same surviving world. The file deliberately avoids proving the full set-theoretic correctness of dedup here; Ring 3 already checks that the profile-level dedup is idempotent and order-insensitive in the real implementation.

survivors: the whole survivor-set algorithm in one line

survivors(bs) = distinct(passingNexts(bs)). That is the entire abstract kernel: first discard failed branches, then collapse duplicate states. Nothing else is needed for the central existential proof. The production code does the same thing on a richer input: a cross-product of candidate states and flattened branches rather than a single list of pre-evaluated booleans.

distinctEmptyIff: dedup does not change emptiness

This is the first explicit theorem in the file. It proves that deduplication cannot create or destroy existence. If a list has at least one element, its deduplicated form also has at least one element; if the list is empty, dedup stays empty. That sounds small, but it is the exact bridge needed to justify that the oracle may decide its verdict based on whether the deduplicated survivor list is empty.

conformantIffSomeBranchPassed: the main verified statement

This theorem instantiates the previous lemma at the oracle level: the survivor list is empty exactly when the list of next-states coming from passing branches is empty. In plain language, some state survives if and only if some branch passed. This is the formally checked core of the verdict logic, and it lines up with the Ring 3 property test that the real implementation is conformant exactly when some branch matches some candidate state.

Bottom line: the file is proving the algebra of existence, not the full business semantics of response checking, violation accumulation, registration, or typed operation dispatch.
Updated assurance story: the verified theorem is now backed by three layers in core: the original oracle properties, direct unit tests for the public kernel functions, and a bridge property asserting that the real ProfileEval agrees with the Stainless model on conformance and deduplicated survivor count.
Saved explanation 2

How the verified mirror maps back to the production oracle

The easiest way to read the file is to follow the real production path first. A caller invokes Spec.allows(op, req, res, profile). If the operation name is not registered, the answer is immediately Verdict.Deviant(UnknownOperation). If it is registered, the method delegates to ProfileEval.allows. That function evaluates each candidate state in the profile, flattens the operation's Outcome tree into atomic branches, keeps only the branches whose checks accept the actual response, computes the next states for those passing branches, dedups them, and finally chooses between Conformant and Deviant.

1. Registration gate

Spec.allows refuses unknown operations before evaluation starts.

2. Flatten outcomes

OutcomeEval.flatten turns Same, Next, and nested OneOf into a plain branch list.

3. Keep passing branches

matches checks whether the actual response satisfies the branch predicate.

4. Compute next states

next applies the branch transition for every surviving candidate.

5. Dedup and verdict

Non-empty survivors yield Conformant; empty survivors yield Deviant.

The key compression step is this: the verified module does not try to re-prove that flattening is correct, that checks accumulate all violations, or that typed operations prevent request and response mismatches. Those guarantees live elsewhere: in the production code, in compile-negative tests, and in the Ring 3 property suite. The new bridge property now adds a direct executable cross-check: it feeds the same generated profiles and responses into real ProfileEval.allows and into the reduced OracleKernel.survivors abstraction, then asserts agreement on conformance and survivor count. The verified module instead isolates the exact piece that is both mathematically central and solver-friendly: survivor existence is equivalent to branch success.

Ring 5 hardening

OracleKernelTests directly checks OutcomeEval.survivors filtering/dedup behavior and the order-insensitive semantics of eqStateProfile.

Ring 6 bridge

OracleModelBridgeTests reduces real branch evaluation to EvalBranch values and checks that model survivors agree with the production verdict and survivor cardinality.

Build wiring

The core project now depends on verified for tests only, while Stainless stays off by default and is turned on explicitly by the ring6 command alias.

Verified mirror

final case class EvalBranch(passed: Boolean, next: BigInt) def passingNexts(bs: List[EvalBranch]): List[BigInt] = bs match { case Nil() => Nil[BigInt]() case Cons(b, t) => if (b.passed) b.next :: passingNexts(t) else passingNexts(t) } def survivors(bs: List[EvalBranch]): List[BigInt] = distinct(passingNexts(bs)) def conformantIffSomeBranchPassed(bs: List[EvalBranch]): Boolean = { survivors(bs).isEmpty == passingNexts(bs).isEmpty }.holds

Production verdict split

def allows[Res, S](..., res: Res, profile: StateProfile[S]): Verdict[S] = val evaluated = profile.toList.map(c => (c, OutcomeEval.flatten(behaviour(c)))) val survivors = OutcomeEval.dedupByEq( evaluated.flatMap((c, branches) => branches.filter(_.matches(res, c)).map(_.next(res, c))) ) survivors match case head :: tail => Verdict.Conformant(StateProfile.of(NonEmptyList(head, tail))) case Nil => Verdict.Deviant(...)

Method by method

What each method proves, and what it intentionally does not prove

passingNexts

Proves by construction: failing branches add nothing; passing branches contribute their next state.

Not proved here: a quantified provenance theorem for every output element.

distinct

Proves by verification: the recursion terminates.

Supports later proof: dedup preserves emptiness.

survivors

Meaning: the abstract survivor-set definition for the verified world.

Purpose: make the final theorem read like the oracle itself.

distinctEmptyIff

Formal theorem: dedup cannot turn non-empty into empty or empty into non-empty.

conformantIffSomeBranchPassed

Formal theorem: survivor existence exactly matches branch success existence.
